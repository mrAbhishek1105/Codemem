# Data Flow & Code Chunking Strategy

## Flow 1: First-Time Indexing

Triggered when user runs `codemem init` or opens a project for the first time.

```
User runs `codemem init`
    │
    ▼
Scan project directory
    │ - Respect .gitignore + .codememignore
    │ - Build file tree
    │ - Detect project type (Node, Python, Rust, etc.)
    │
    ▼
Generate project summary
    │ - Package.json / requirements.txt / Cargo.toml analysis
    │ - Directory structure map
    │ - Entry point detection
    │ - Framework detection (React, Django, etc.)
    │
    ▼
Chunk each file
    │ - Parse into logical units (functions, classes, modules)
    │ - Attach metadata (file path, language, imports, exports)
    │ - Create context windows (chunk + surrounding context)
    │
    ▼
Generate embeddings (local model)
    │ - Run all-MiniLM-L6-v2 via ONNX runtime
    │ - Batch processing (64 chunks at a time)
    │ - ~500 chunks/second on modern CPU
    │
    ▼
Store in ChromaDB
    │ - Vectors + metadata + raw text
    │ - Build dependency graph
    │ - Cache file hashes for change detection
    │
    ▼
Start file watcher
    │ - Monitor for changes
    │ - Ready to serve queries
    │
    ▼
Output: "Indexed 847 files (12,340 chunks) in 23s. Ready."
```

### Timing Estimates (First Index)

| Project Size | Files | Chunks | Index Time | DB Size |
|-------------|-------|--------|------------|---------|
| Small (starter) | ~50 | ~500 | ~3s | ~5MB |
| Medium (SaaS app) | ~200 | ~3,000 | ~10s | ~25MB |
| Large (monorepo) | ~2,000 | ~30,000 | ~60s | ~200MB |
| Huge (enterprise) | ~10,000 | ~150,000 | ~5min | ~1GB |

## Flow 2: Prompt Query (Normal Usage)

Triggered every time the user sends a prompt to their AI assistant.

```
User types prompt in IDE
  "Fix the login validation bug on the signup form"
    │
    ▼
IDE adapter intercepts (or MCP tool is called)
    │
    ▼
Sidecar receives query
    │
    ▼
Query analysis
    │ - Extract key terms: "login", "validation", "signup", "form"
    │ - Identify intent: bug fix
    │ - Check recent changes context
    │
    ▼
Vector search (ChromaDB)
    │ - Semantic search with query embedding
    │ - Top-K retrieval (default K=10)
    │ - Metadata filtering (prioritize relevant file types)
    │
    ▼
Context assembly
    │ - Rank retrieved chunks by relevance score
    │ - Pull in dependency context (imports, types, callers)
    │ - De-duplicate overlapping chunks
    │ - Fit within token budget (default: 4,000 tokens)
    │
    ▼
Return assembled context to AI
    │ - Relevant code chunks (3-8 files typically)
    │ - Project structure summary (cached)
    │ - Recent changes context
    │
    ▼
AI generates response using focused context
    │ - 2,000-4,000 tokens instead of 50,000+
    │
    ▼
Output tracking
    │ - Log tokens saved
    │ - Update stats.json
```

## Flow 3: Incremental Re-Indexing

Triggered automatically when files change (file watcher detects changes).

```
File watcher detects change
  (user saves file, git pull, etc.)
    │
    ▼
Compute file hash
    │ - Compare with stored hash in file-hashes.json
    │ - If unchanged (e.g., just touched): skip
    │
    ▼
Re-chunk the changed file
    │ - Parse into logical units
    │ - Compare chunks with previous version
    │ - Identify: new chunks, modified chunks, deleted chunks
    │
    ▼
Update vectors
    │ - Delete old chunk vectors from ChromaDB
    │ - Generate new embeddings for changed chunks
    │ - Insert new vectors
    │ - Takes ~50ms per file
    │
    ▼
Update metadata
    │ - Update file-hashes.json
    │ - Update dependency-graph.json if imports changed
    │ - Update project-summary.json if structure changed
    │
    ▼
Ready for next query (no restart needed)
```

### Change Detection Strategy

```
Event: File modified (e.g., src/auth/login.ts)
    │
    ├── Hash check: content actually changed?
    │   ├── No  → skip (just a touch/save without changes)
    │   └── Yes → continue
    │
    ├── Debounce: wait 500ms for more changes
    │   └── (handles rapid saves, IDE auto-format, etc.)
    │
    ├── Batch processing: group all changes in window
    │   └── (handles git checkout switching 100+ files)
    │
    └── Incremental update: only re-embed changed chunks
        └── (NOT full re-index — just the diff)
```

## Flow 4: AI Provider Switch

Triggered when user changes their AI provider configuration.

```
User runs: codemem config --ai claude
    │
    ▼
Update config.json
    │ - Set default AI provider
    │ - No re-indexing needed!
    │
    ▼
Vector DB unchanged
    │ - Embeddings are AI-provider-independent
    │ - Generated by LOCAL embedding model
    │ - Same vectors work with ANY AI
    │
    ▼
Next prompt: new AI gets context from same memory
    │ - Zero token waste
    │ - Zero re-reading
    │ - Instant switch
```

## Code Chunking Strategy

### Chunking Principles

1. **Chunk by semantic units, not line count**
   - A function is one chunk (regardless of length)
   - A class is one chunk (with methods as sub-chunks)
   - Top-level imports/exports are a separate chunk
   - Large files get multiple chunks; small files may be one chunk

2. **Include context in every chunk**
   - File path (always)
   - Import statements (always)
   - Parent class/module name (if applicable)
   - Type definitions used by the chunk (if available)
   - Export status (is this chunk exported/public?)

3. **Respect language-specific boundaries**
   - Different parsers for different languages
   - Use tree-sitter for AST-based chunking (best accuracy)
   - Fall back to regex-based chunking for unsupported languages

### Chunk Structure

```json
{
  "id": "src/auth/login.ts::validateCredentials",
  "type": "function",
  "file_path": "src/auth/login.ts",
  "language": "typescript",
  "name": "validateCredentials",
  "content": "async function validateCredentials(email: string, password: string): Promise<AuthResult> {\n  // ... function body\n}",
  "context": {
    "imports": ["import { AuthResult } from '../types'", "import { hash } from '../utils/crypto'"],
    "parent": null,
    "exports": ["export { validateCredentials }"],
    "called_by": ["src/routes/auth.ts::loginHandler"],
    "calls": ["src/utils/crypto.ts::hash", "src/db/users.ts::findByEmail"]
  },
  "metadata": {
    "start_line": 24,
    "end_line": 45,
    "last_modified": "2025-03-28T10:30:00Z",
    "content_hash": "a1b2c3d4...",
    "token_count": 180
  }
}
```

### Chunking Rules by Language

| Language | Parser | Chunk Boundaries |
|----------|--------|-----------------|
| TypeScript/JavaScript | tree-sitter-typescript | Functions, classes, interfaces, type aliases, top-level const |
| Python | tree-sitter-python | Functions, classes, top-level assignments |
| Rust | tree-sitter-rust | fn, impl blocks, struct, enum, trait |
| Go | tree-sitter-go | Functions, types, interfaces |
| Java/Kotlin | tree-sitter-java | Methods, classes, interfaces |
| C/C++ | tree-sitter-c | Functions, structs, classes |
| Ruby | tree-sitter-ruby | Methods, classes, modules |
| PHP | tree-sitter-php | Functions, classes |
| Fallback (any) | Regex-based | Function/class patterns, blank-line separation |

### Chunk Size Limits

- Minimum: 50 tokens (skip trivial getters, one-liners)
- Target: 100-500 tokens (sweet spot for embedding quality)
- Maximum: 1,000 tokens (split larger functions into logical blocks)
- If a function exceeds 1,000 tokens: split at logical breakpoints
  (conditionals, loop boundaries, comment blocks)

## Context Assembly Strategy

When the AI sends a query, CodeMem doesn't just return raw chunks. It assembles
a smart context package:

### Assembly Algorithm

```
Input: user query + retrieval results (top-K chunks)

Step 1: Primary retrieval
  - Embed the query using local model
  - Search ChromaDB for top 10 similar chunks
  - Each chunk has a relevance score (0.0 - 1.0)

Step 2: Dependency expansion
  - For each retrieved chunk, check dependency graph
  - Pull in: imported types, called functions, parent classes
  - But only if they aren't already in the result set
  - Limit: max 5 additional dependency chunks

Step 3: Recency boost
  - If any recently modified files are relevant, boost their score
  - "Recently modified" = changed in last 24 hours
  - Boost factor: 1.2x relevance score

Step 4: Token budget fitting
  - Sort all chunks by final relevance score
  - Add chunks until token budget is reached (default: 4,000)
  - Always include: project summary header (~200 tokens)
  - Always include: file tree context (~100 tokens)

Step 5: Format output
  - Group chunks by file
  - Add file path headers
  - Add "..." markers for skipped sections
  - Add dependency annotations
```

### Output Format (Sent to AI)

```
## Project: my-saas-app (Node.js/TypeScript, React frontend)

### Relevant files for your query:

--- src/auth/login.ts (lines 24-45) ---
import { AuthResult } from '../types';
import { hash } from '../utils/crypto';

async function validateCredentials(email: string, password: string): Promise<AuthResult> {
  const user = await findByEmail(email);
  if (!user) return { success: false, error: 'User not found' };
  const valid = await hash.compare(password, user.passwordHash);
  return { success: valid, error: valid ? null : 'Invalid password' };
}

--- src/components/SignupForm.tsx (lines 15-62) ---
// ... relevant component code ...

--- src/types/auth.ts (dependency) ---
export interface AuthResult {
  success: boolean;
  error: string | null;
  token?: string;
}

### Recent changes (last 24h):
- src/auth/login.ts: modified validateCredentials (added rate limiting)
- src/components/SignupForm.tsx: added email validation
```

## Semantic Context Layer (Differentiator)

Beyond just code, CodeMem stores enriched context that evolves over time:

### What Gets Stored

1. **Code chunks** — the actual source code as vectors
2. **Dependency graph** — which files import/call which
3. **Change history** — what changed, when, and in what pattern
4. **Error patterns** — if AI-generated code caused errors, record them
5. **Architecture decisions** — detected patterns (MVC, microservices, etc.)
6. **Co-change clusters** — files that tend to change together
7. **API surface** — exported functions, routes, endpoints

### Co-Change Detection

```
Track: When file A changes, do files B and C also change?

After 5+ observations of A+B changing together:
  → Record as co-change cluster
  → When AI modifies A, automatically include B in context
  → Prevents the "you changed the handler but forgot the test" problem

Example:
  src/routes/users.ts ←→ src/tests/users.test.ts (92% co-change)
  src/models/User.ts ←→ src/types/user.d.ts (87% co-change)
```
